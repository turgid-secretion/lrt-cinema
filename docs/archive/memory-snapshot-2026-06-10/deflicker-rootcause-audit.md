**B2 deflicker "~3×" — REFUTED (commit 4de7a7a, doc docs/research/deflicker-rootcause-audit.md, PR #47).** The 250-frame findings claimed the deflicker is applied ~3× too weak (the #1 north-star lever) + added `--deflicker-scale` to dial in ~2.7×. **The audit refutes it: there is no under-application. Keep `--deflicker-scale 1.0` (it's correct, not just byte-exact).**

**NO BUG (verified):** parser reads `crs:LocalExposure2012` correctly; the deflicker **mask is effectively full-frame** (a "Mask/Gradient" positioned OFF-frame — `FullY=-0.3/ZeroY=-0.31` both above y=0, `CorrectionAmount=1.0`, `MaskValue=1.0`, named "LRT: don't alter!" = LRT's trick to apply a *global* exposure via the local-correction mechanism); CLI applies it 1:1 (`apply_lrt_mask_offsets`→`exposure_ev += EV`→`2^EV`). In the first-250 window the deflicker is the **ONLY non-zero op** (global Exposure/Contrast/Highlights/Shadows + the **HG and Global** mask components all 0.0 across all 250 — I initially omitted HG/Global; checked, they're 0).

**KEEP 1.0 ON THREE INDEPENDENT LEGS:** (1) **scalar-gain conflation** — the "~3×" was back-solved from a no-offset gain `g=(o·t)/(o·o)`, which has no single correct value for a tone-SHAPE difference (it drifts as the scene moves along the curve), so it manufactures a fake deflicker factor; (2) **short-term BY DESIGN** — LRT's Visual Deflicker "smooths short-term flicker, leaves mid/long-term changes alone" (VISUAL_WORKFLOW.md), so the LONG-term gain drift is *deliberately not* deflicker-corrected (domain-independent, no test); (3) **linear preview test** — `.lrt/visual/*.lrtpreview` are LRT's pre-deflicker developed JPEGs (proven pre-deflicker: their short-term flicker is cancelled by `d`); finding the `k` that flattens the high-freq flicker of `log2(P_i)+k·d_i` in **linear** luminance gives `k*≤1` and **scaling ≥2× monotonically WORSENS flicker in EVERY high-pass window** (w∈{5,7,11,15,21}: w=7 → 5.6e-4@k1 → 7.4e-4@k2 → 9.6e-4@k2.7). (Lead with that ordering, not the noisy ~0.5 point estimate; keep the LRT-authored 1.0.)

**THE GENERAL LESSON (corrected — NOT a gamma artifact):** **a scalar no-offset per-frame gain conflates EXPOSURE with TONE-SHAPE**, so a fixed tone-curve difference reads as a drifting gain and back-solves into a fake deflicker factor. NB the gain table is **already linear** — the tool linearizes (`eotf_sRGB`) before the fit, and I VERIFIED linear≈gamma gain to ~1% (1.0019→1.1391 linear vs 1.0101→1.1324 gamma). So the defect is the *scalar gain metric*, NOT the bit depth (an earlier draft of this note wrongly called it a "gamma-domain artifact" — corrected). Don't summarise an LRT-vs-ours comparison with a scalar gain when the suspected difference is a curve.

**TWO CONFIDENCE LEVELS (don't fuse them):** (a) ESTABLISHED — scaling refuted, keep 1.0 (3 legs). (b) INFERENCE (pending §11's direct extraction) — the residual long-term drift is the PV2012 tone-curve-shape gap (lrt-jpg-northstar-baseline; DECISIONS §11). Support = the **within-frame tone tilt** (a single frame's per-pixel LRT/ours linear ratio is luminance-dependent, e.g. frame 1: 0.66 shadows → 0.93 highlights — ONE scene, no cross-frame confound) + short-term-by-design. NOT proven: the cross-frame **frame-independence** transfer-curve test came back **inconclusive — scene-confounded** (a day→evening timelapse renders different scenes, so a luminance-binned per-pixel mapping mixes content/colour and can't isolate a fixed tone curve; rendered 3 current frames to check — std 0.12, neither confirmed nor refuted). Guardrail: scalar-gain conflation explains why the gain CAN'T separate exposure from tone — it does NOT by itself PROVE tone-shape. §11 settles it.
