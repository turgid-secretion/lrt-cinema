# LRTimelapse pixel pipeline and inter-process architecture

> Scope: what LRTimelapse renders by itself versus what it delegates to
> Lightroom Classic, the file artifacts produced by each path, and the
> communication channel between the two applications.
>
> Sources: LRT official tutorials and workflow pages, archive of release
> announcements 7.0–7.5.3, forum posts authored by Gunther Wegner, and
> empirical inspection of the user's LRT 7.5.3 sequence on 2026-05-22.
> Every claim below is either sourced or tagged `OBSERVED, NOT
> DOCUMENTED`. Speculative inference is replaced by `STATUS: UNKNOWN`.

## Two-path architecture

LRTimelapse 7.x exposes two end-to-end rendering paths. Both share the
same XMP-authoring front end (Keyframes Wizard → Auto Transition →
Visual Previews → Visual Deflicker → Holy Grail Wizard); they diverge
at the "Export & Render" step.

- **Internal path.** "Export & Render (internal)" produces the
  intermediate frames inside LRT itself. The documented limit:
  *"the internal export will always create 8 bit sRGB intermediary JPG
  files"* and *"the quality will be inferior to the quality provided by
  the Lightroom Export via the LRTExport plugin"*
  (https://lrtimelapse.com/workflow/internal-workflow/). This is the
  path our project replaces with darktable to obtain 16-bit linear
  TIFFs.
- **Lightroom path.** "Export & Render (Lightroom)" hands off via the
  bundled LRTExport plugin. The user first imports the sequence into
  Lightroom Classic (drag-and-drop from LRT's UI), runs *Metadata →
  Read Metadata from Files* so LR picks up LRT-written XMPs, then
  selects an LRT-provided export preset such as "JPG (4k)". After the
  LR export completes *"Lightroom automatically opens the render dialog
  in LRTimelapse"* and LRT assembles the LR-rendered intermediate
  frames into the final video
  (https://lrtimelapse.com/tutorial/basic/complete/).

The interface between LRT and Lightroom is XMP files plus drag-and-drop
plus the export plugin's IPC. The LRT documentation states
*"LRTimelapse communicates with the raw converter or Lightroom via
these files"* (referring to per-frame XMP sidecars). There is no
documented evidence of LRT calling Lightroom's SDK programmatically;
the user-facing flow is manual drag-and-drop initiated import.

## Preview pipeline (what `.lrtpreview` actually contains)

Three artifacts live under `.lrt/` for every initialized sequence;
their formats are observed empirically (see DISK_LAYOUT.md for SHA-256
values and exact file metadata):

| Folder | Format | Geometry | Quality | When generated |
|---|---|---|---|---|
| `.lrt/previews/` | JPEG (JFIF, no EXIF, no ICC) | 640×424 | baseline, YCbCr 4:2:0 | Once per frame during *Initialize* |
| `.lrt/visual/` | JPEG (JFIF, no EXIF, no ICC) | 1024×684 | baseline, YCbCr 4:2:0 | Per keyframe on *Visual Previews* (keyframes only), per frame on *Visual Previews → All Frames* |
| `.lrt/proxy/` | DNG 1.4 (lossy JPEG, linear-raw photometric, 256×171 tile, full embedded CRS schema) | 1024×684 | DNG with embedded 1024-wide proxy + 256-wide thumbnail | Per keyframe when *Proxy* generation is requested |

The 640-wide previews carry no ICC profile, no EXIF tags, no XMP
packet — only the JFIF header and the JPEG image data. ImageMagick's
`-verbose` reports `Colorspace: sRGB` because that is its fallback for
untagged JPEGs (`identify` defaults to sRGB when the JFIF marker is
present and no profile is embedded), not because the file carries an
sRGB tag. The 1024-wide visual previews follow the same pattern.
`OBSERVED, NOT DOCUMENTED.`

The `.proxy` file extension is misleading. `exiftool` and `file -i`
identify the magic bytes as TIFF/DNG produced by *Adobe DNG Converter
18.2.2 (Macintosh)*, with `Profile Look Table Encoding: sRGB` and a
1024×684 preview image embedded. This is the bombshell observation
in this section: **LRT does not run a homebrew RAW demosaic for its
preview pipeline. It bundles Adobe DNG Converter and shells out to it
to produce the proxies.** The `Software` EXIF tag reads `Adobe DNG
Converter 18.2.2 (Macintosh)` on the user's sample, which was
generated by LRT 7.5.3, not by a manual user step.

This means the `.lrt/proxy/*.proxy` files are produced by Adobe code,
and the per-frame XMP that LRT writes alongside them (when a keyframe
has been initialized — see DISK_LAYOUT.md) carries the full Adobe CRS
schema including `crs:CameraProfile`, `crs:LensProfileEnable`,
`crs:HasCrop`, etc. The 640- and 1024-wide JPEG previews are presumed
to be generated from the DNG proxy as a downstream rendering step
inside LRT, but the LRT documentation does not describe this
explicitly. `STATUS: UNKNOWN whether the JPEG previews are rendered by
LRT directly from the DNG proxy, or whether DNG Converter writes them
as a side-effect during proxy generation.`

## Does the LRT preview honor `crs:LensProfileEnable` / `crs:HasCrop`?

This was the inferential trap that prompted this entire reference. The
project had been operating on the assumption that the LRT preview
"almost certainly" applied LR-style lens correction. The honest answer
on review of the sources:

- The LRT documentation makes no statement on this.
- The user's keyframe XMPs (e.g. `.lrt/proxy/DSC_5059.xmp`) carry
  `crs:LensProfileEnable="0"` and `crs:HasCrop="False"` (no profile
  enabled, no crop). So our sample cannot empirically discriminate
  whether LRT's preview honors these flags — it would only matter on a
  sequence where the user had toggled them on.
- `STATUS: UNKNOWN.` Reproducing this would require a keyframe XMP
  with `crs:LensProfileEnable="1"` and a known-distorted lens, then
  comparing the `.lrt/visual/` JPEG against a hand-rendered
  uncorrected reference. Recommended as future-work; not in scope for
  this reference.

## Auto Transition is XMP-authoring, not pixel-rendering

Auto Transition operates on metadata, not pixels. Per the LRT internal
workflow tutorial: *"Auto Transition calculates the developments for
all images between the keyframes."* In disk terms, this means LRT
writes a per-frame XMP sidecar for every frame in the sequence with
interpolated `crs:Exposure2012` and other animated fields. The Visual
Previews step then re-renders the `.lrt/previews/*.lrtpreview` JPEGs
using the updated per-frame XMPs.

See AUTO_TRANSITION.md for the interpolation model (what spline
shape, what fields are animated, what is held constant).

## What our project depends on from this pipeline

- The per-frame XMPs LRT writes after Auto Transition are our IR
  source. Parser must accept the schema documented in XMP_SCHEMA.md.
- The internal-render path (8-bit sRGB JPG) is what we are
  substituting darktable for, to obtain 16-bit linear cinema
  intermediates. See `docs/V03_PLAN.md` for the render fidelity track.
- We do *not* depend on LRT's preview pipeline at runtime. Our
  understanding of it matters only because it shaped LRT's design
  assumptions (8-bit sRGB output, Adobe DNG Converter dependency) that
  we are routing around.
